## 📚 阶段1：夯实基础

**目标**：深入理解 Agent 的核心循环，掌握 OpenAI SDK 的更多细节，为后续扩展打下基础。  
（以下要点供复习和巩固，每个都包含思考题）

### 学习要点

1. **消息结构再理解**  
   - 回顾 `system`、`user`、`assistant`、`tool` 四种角色的交互逻辑。  
   - 实操：手动构造一个包含多轮对话和工具调用的 messages 数组，用本地模型（如通过 Ollama 兼容接口）模拟一次完整的工具调用流程（不实际执行工具，只看返回的 tool_calls）。  
   ❓ **思考题**：如果模型返回的 `tool_calls` 中 `id` 字段缺失，你应该如何构造 tool 消息回传结果？OpenAI 规范对此有要求吗？

2. **函数调用（Tool Calling）精讲**  
   - 学习 JSON Schema 的详细写法（参数类型、枚举、嵌套对象、必填项）。  
   - 实操：为几个虚构工具（如 `get_stock_price`、`send_email`）编写 JSON Schema，并通过本地模型测试模型是否能正确调用这些工具。  
   ❓ **思考题**：假设你的工具参数中包含一个嵌套对象 `{ "address": { "city": "北京", "street": "长安街" } }`，如何编写 JSON Schema 来描述它？模型返回的参数格式必须完全匹配吗？

3. **并行工具调用处理**  
   - 理解模型同时返回多个 tool_calls 的场景，以及每个调用都必须通过唯一的 ID 回传结果。  
   - 实操：构造一个同时问两个独立问题（如“北京天气和 123*456”）的请求，编写代码解析返回的多个工具调用，分别执行后再将所有结果打包回传给模型。  
   ❓ **思考题**：如果并行调用中的某个工具执行失败，你是应该终止整个流程还是只回传该工具的错误信息？模型可能会如何处理这种部分失败的情况？

4. **流式输出（Streaming）**  
   - 学习如何用 `stream=True` 实现逐字输出，并捕捉流中的 tool_calls 事件。  
   - 实操：修改之前的单次调用代码，改为流式模式，将文本输出实时打印，并正确处理流式出现的工具调用（提示：部分 SDK 会在流中通过多个 chunk 拼出完整的 tool_calls）。  
   ❓ **思考题**：在流式模式下，如果模型在生成文本中途决定调用工具，你如何判断何时停止输出文本并开始处理工具调用？流式响应的数据结构有何特点？

5. **错误处理与重试**  
   - 掌握捕获 API 异常（如超时、速率限制）并实现指数退避重试。  
   - 学习如何处理工具执行中的异常（比如函数抛出错误），并将错误信息构造为 tool 消息返回给模型，让模型重新尝试或给出合理解释。  
   - 实操：在现有 Agent 中加入 try-catch 逻辑，故意让一个工具抛出异常，观察模型能否根据错误信息重新调用或给出提示。  
   ❓ **思考题**：如果工具返回的错误信息过长，会占用大量 token，你有什么策略来压缩或摘要错误信息？应该完全信任模型能根据错误信息自我修正吗？

### 🛠️ 实践项目回顾：增强版工具助手

- 升级你之前的天气/计算器 Agent，要求：
  - 支持流式输出（用户体验优化）。
  - 支持并行工具调用（例如同时问“上海天气和 99*88”）。
  - 具备基本的错误重试机制（API 超时重试 + 工具执行异常反馈）。
  - 将整个对话历史（短期记忆）保存在一个列表中，支持滑动窗口（限制最近 N 轮），为阶段2打好基础。

> **代码结构参考**：  
> - `main.py`：Agent 主循环入口  
> - `core/stream.py`：流式处理逻辑  
> - `core/memory.py`：短期记忆管理（滑动窗口）  
> - `tools/`：存放天气、计算器等工具，已实现自动发现  
> - `.env`：配置 API 地址、模型名称（如 `qwen3.5:4b`）

## 🧠 阶段2：构建记忆系统（2-3周）

**目标**：让 Agent 拥有“短期对话管理”和“长期事实记忆”，能记住用户偏好、历史关键信息。  
本阶段将引入本地向量数据库和嵌入模型，请先选择以下组件：

- **向量数据库选项**：Chroma（轻量级，嵌入式）、FAISS（本地，需管理索引）、LanceDB（新兴，Rust 核心）。建议先用 Chroma，上手简单。
- **嵌入模型选项**：`text-embedding-3-small`（OpenAI，需API，不推荐本地）、`bge-small-zh`（国产，本地运行）、`m3e-base`（国产，本地运行）。建议先用 `bge-small-zh` 或 `m3e-base` 通过 `sentence-transformers` 或 `FlagEmbedding` 加载。

确认选择后，我们开始学习。

### 学习要点（循序渐进）

1. **短期记忆管理**  
   - 理解短期记忆即对话历史，但 token 长度有限，需要滑动窗口管理。  
   - 实操：在 `core/memory.py` 中编写一个 `ConversationBuffer` 类，保存最近 K 轮对话（每轮包括 user 和 assistant 消息），并提供 `get_messages()` 方法返回完整 messages 列表供模型调用。可以简单限制轮数，也可以按 token 数截断。  
   ❓ **思考题**：如果使用按 token 截断，如何估算 token 数量？对于中文模型，通常每个汉字算几个 token？你会选择在每次添加消息时截断，还是每次获取时截断？

2. **长期记忆与向量数据库入门**  
   - 学习为什么需要向量库：语义搜索比关键词搜索更适合模糊记忆。  
   - 实操：安装 Chroma，学习如何创建集合、添加文档（包含文本和元数据）、将文本转为向量（使用选定的本地嵌入模型）。写一个脚本，添加几条记忆（如“用户喜欢吃辣”），然后进行相似度搜索，观察结果。  
   ❓ **思考题**：向量检索的结果受嵌入模型影响很大。如果用户说“我爱吃川菜”，和“我喜欢麻辣口味”在语义上相似吗？你的嵌入模型能捕捉到这种相似性吗？如何验证？

3. **记忆的存取流程**  
   - **存储**：设计一个 `extract_facts(text)` 函数，可调用模型提取对话中值得记忆的事实，或者用简单规则（如“我叫XX”的句式）。将提取的事实存入向量库，同时标记用户 ID 和会话 ID。  
   - **检索**：每次用户新提问前，将用户输入转为向量，从向量库召回最相关的 3-5 条记忆，将这些记忆文本拼接到 system prompt 或作为新的 user 消息前缀。  
   - 实操：在现有的 Agent 循环中，集成上述存取逻辑。测试：第一次告诉 Agent “我喜欢吃川菜”，第二次问“推荐什么菜”，看 Agent 是否能推荐辣菜。  
   ❓ **思考题**：如果用户同时提到多个事实（例如“我叫小明，住在北京，喜欢运动”），你应该存储为一条记忆还是多条？哪种方式更有利于后续检索？

4. **对话压缩与摘要**  
   - 当短期记忆过长（比如超过 10 轮），自动触发摘要生成，用摘要替换掉旧的对话，减少 token 占用。  
   - 实操：实现一个 `summarize_conversation(history)` 函数，调用模型生成摘要，然后将摘要作为一条 system 消息插入，同时清空旧的对话历史。在 Agent 运行中，每次检查历史长度，超过阈值就进行压缩。  
   ❓ **思考题**：摘要可能会丢失细节。如果用户后来问起摘要中已忽略的细节，Agent 该如何应对？是否有必要同时保留原始对话的向量索引？

### 🛠️ 实践项目：带个人记忆的助手

- 实现一个完整的记忆系统，要求：
  - 使用 Chroma 作为向量存储，每个用户有自己的记忆空间（通过元数据区分）。
  - 短期记忆采用滑动窗口（例如最近 10 轮）。
  - 每次用户输入前，从向量库检索相关记忆，拼入 prompt。
  - 每次对话中，通过简单规则或模型提取新事实，异步存入向量库（注意不要阻塞主流程）。
  - 当短期记忆轮数超过 12 轮时，自动生成摘要并替换早期对话。
  - 测试场景：多轮交互后，关闭程序再重启，检查 Agent 是否还记得之前的偏好（例如“我叫小明”、“我喜欢运动”）。
- **代码位置建议**：
  - `core/memory.py`：`ConversationBuffer`、`VectorMemory` 类。
  - `core/agent.py`（可能需要新建）或 `main.py` 中集成记忆逻辑。
  - `config.py`：配置向量库路径、嵌入模型名称等。

## 🔧 阶段3：工具调用进阶（2-3周）

**目标**：让 Agent 能动态使用工具、组合工具，并处理复杂的工具调用场景，同时兼顾安全和健壮性。  
本阶段将基于你已有的 `tools/` 模块进行扩展，利用 `tools/__init__.py` 自动发现工具。

### 学习要点（循序渐进）

1. **动态工具注册**  
   - 抛弃硬编码工具列表，实现一个工具注册表（`tools_dict`），用装饰器或配置文件动态加载函数。  
   - 实操：在 `tools/__init__.py` 中编写一个 `@tool` 装饰器，将函数名和函数对象注册到全局字典，并提供工具描述的 JSON Schema（可以先手动写 Schema，后续自动生成）。  
   ❓ **思考题**：如果两个工具同名，注册时应该覆盖还是报错？你的注册表如何支持工具别名（如 `weather` 和 `get_weather` 指向同一个函数）？

2. **工具描述的自动生成**  
   - 学习如何使用 `inspect` 模块或 `pydantic` 从函数签名和文档字符串自动生成 JSON Schema。  
   - 实操：为之前写的几个工具函数加上类型注解和文档字符串，然后编写一个函数 `generate_schema(func)`，返回符合 OpenAI 格式的工具定义。将动态注册和自动生成结合起来。  
   ❓ **思考题**：如果函数参数是 `**kwargs` 或未注解，自动生成 schema 时应该怎么处理？是否可以为这类参数提供 fallback 机制？

3. **复杂工具参数处理**  
   - 学习如何描述嵌套对象、数组等复杂参数（如 `{ "people": [{"name": "张三", "age": 30}] }`）。  
   - 实操：设计一个工具 `batch_send_emails(recipients: List[dict])`，参数为人员列表，编写其 JSON Schema，并用模型测试能否正确生成调用参数。同时，处理模型返回参数缺失或类型错误的情况（如用默认值或报错回传）。  
   ❓ **思考题**：模型生成的参数可能不符合 schema（例如缺少必填字段），你的工具执行器应该直接报错还是尝试智能补全？如果补全，补全的逻辑依据是什么？

4. **工具调用结果的反馈优化**  
   - 如果工具返回错误（如文件不存在、网络超时），如何构造 tool 消息让模型重新尝试或给出合理解释。  
   - 实操：修改工具函数，在特定条件下抛出异常或返回错误信息，观察模型是否能根据错误信息调整参数再次调用，或向用户说明。  
   ❓ **思考题**：工具返回的错误信息可能很长（如堆栈跟踪），直接返回会浪费 token。你有什么策略来精简错误信息，同时保留足够上下文让模型理解？

5. **权限与安全**  
   - 学习如何限制工具可访问的资源（如文件系统沙箱、网络白名单），对敏感工具添加用户确认机制。  
   - 实操：实现一个“发送邮件”工具，要求在执行前询问用户确认（可通过命令行输入 y/n），只有在确认后才真正发送（模拟）。同时，限制文件读写工具只能操作特定目录。  
   ❓ **思考题**：如果用户通过对话诱导模型调用敏感工具（如“帮我删除所有文件”），你的护栏机制如何防止？应该由模型判断还是由工具执行层拦截？

### 🛠️ 实践项目：多功能办公助手

- 构建一个动态工具集成的 Agent，包含：
  - 一组办公工具：发邮件（模拟）、创建日历事件、搜索网络（模拟）、读写文件（限定沙箱目录）。
  - 使用装饰器自动注册所有工具，并从函数签名自动生成工具描述。
  - 支持复杂参数：例如创建日历事件时需要传入 `attendees` 数组。
  - 包含错误处理和权限控制：文件操作只能在指定目录；发送邮件前要求用户确认。
  - 测试一个复合指令：“给张三、李四发邮件，说我们明天下午3点开会，并把会议纪要保存到 `meeting_notes.txt`”。Agent 应能分解任务，依次调用相应工具。
- **代码位置建议**：
  - `tools/__init__.py`：完善 `@tool` 装饰器，添加注册表和 schema 生成。
  - `tools/email.py`、`tools/calendar.py`、`tools/file.py` 等新工具文件。
  - `core/tool_executor.py`（新建）负责调度工具调用、处理错误和权限。

## 🚀 阶段4：增强能力（3-4周）

**目标**：让 Agent 具备规划、多智能体协作等高级能力，能处理更复杂的任务。

### 学习要点（循序渐进）

1. **规划（Planning）**  
   - 学习 ReAct 模式的优化，引入显式的规划步骤。  
   - 实操：实现一个“规划器”模块，在复杂任务开始时，让模型先生成一个步骤列表（例如用 JSON 格式输出 `{"steps": ["step1", "step2"]}`），然后逐步执行。每一步执行后更新进度，直到所有步骤完成。  
   ❓ **思考题**：如果模型生成的步骤列表不完整或顺序错误，你的规划器应该如何应对？是否需要加入“重新规划”机制？

2. **子任务分解与执行**  
   - 将大任务拆分为多个小任务，可分配给不同工具或子 Agent。  
   - 实操：扩展规划器，使其能够根据步骤调用不同工具，并处理步骤间的依赖（如步骤2需要步骤1的结果）。  
   ❓ **思考题**：如果某个子任务执行失败，规划器应该终止整个任务还是尝试替代方案？如何让模型参与决策？

3. **多智能体协作（Multi-Agent）**  
   - 学习智能体交接（Handoff）模式：一个 Agent 将任务委托给另一个专业 Agent。  
   - 实操：创建三个专业 Agent（天气 Agent、日历 Agent、邮件 Agent），每个 Agent 有自己的工具集。实现一个总控 Agent，它有一个工具 `transfer_to_agent(agent_name)`，通过这个工具实现任务交接。维护一个 Agent 注册表，根据用户请求类型动态选择。  
   ❓ **思考题**：交接时上下文信息如何传递？是让总控 Agent 直接传递对话历史，还是每个子 Agent 都共享同一个长期记忆系统？

4. **状态管理与对话树**  
   - 复杂任务可能需要维护状态（如多步骤表单填写）。使用有限状态机（FSM）管理 Agent 状态。  
   - 实操：设计一个“预订会议室”的多轮对话流程，Agent 需要依次收集“日期、时间、参会人数、会议室设备要求”。用 FSM（例如 `transitions` 库或手写状态类）控制对话流转，每个状态对应一个处理函数。  
   ❓ **思考题**：如果用户在对话中途改变主意（例如说“算了，不订了”），你的 FSM 如何优雅地退出当前流程并重置状态？

5. **护栏（Guardrails）**  
   - 学习输入护栏：检查用户输入是否安全、是否在任务范围内；输出护栏：检查模型输出是否合规，防止注入攻击。  
   - 实操：添加一个输入检查函数，拒绝包含敏感词或明显越权的请求（如“删除所有文件”）。输出检查可以拦截包含危险代码的回复。  
   ❓ **思考题**：护栏可能会误伤正常请求（如“给我讲个关于黑客的故事”包含“黑客”一词）。如何平衡安全性和用户体验？是否可以引入白名单机制？

### 🛠️ 实践项目：团队协作助手

- 实现一个多 Agent 协作系统，要求：
  - 有一个总控 Agent，它通过 `transfer_to_agent` 工具将任务分发给天气、日历、邮件三个专业 Agent。
  - 支持简单的规划能力：当用户说“安排明天的会议并通知大家”，总控先生成计划（“创建日历事件，然后发送邮件”），然后依次交给日历 Agent 和邮件 Agent 执行。
  - 在预订会议室的场景中，使用 FSM 管理多轮信息收集。
  - 添加基本的护栏：检查输入是否包含恶意指令，输出中是否包含敏感信息。
  - 测试用例：1）“明天下午3点开会，通知张三和李四”；2）“帮我预订一个后天上午10点能容纳10人的会议室，需要有投影仪”。
- **代码位置建议**：
  - `core/planner.py`：规划器逻辑。
  - `core/agent_registry.py`：管理多 Agent。
  - `core/fsm.py`：状态机实现（可单独模块）。
  - `core/guardrails.py`：输入输出检查。

## 🏭 阶段5：工程化与部署（2-3周）

**目标**：将 Agent 打包成可靠的服务，支持多用户并发，并具备可观测性。

### 学习要点（循序渐进）

1. **会话管理**  
   - 学习如何为每个用户隔离对话历史和记忆（用 user_id 作为命名空间）。  
   - 实操：在阶段4的代码基础上，修改向量库和对话缓存，加入 `user_id` 字段。API 接口接收 `user_id` 参数，根据用户 ID 加载对应的记忆和对话历史。  
   ❓ **思考题**：如果用户数量巨大，每个用户的记忆都独立存储，如何保证检索效率？是否需要引入缓存层？

2. **API 封装**  
   - 使用 FastAPI 将 Agent 暴露为 RESTful API。  
   - 实操：设计请求体包含 `user_id` 和 `message`，响应为流式或一次性返回。实现一个简单的 `/chat` 端点，调用 Agent 主循环并返回结果。  
   ❓ **思考题**：流式响应如何用 FastAPI 实现？你可能会用到 `StreamingResponse`，但如何与异步 Agent 循环配合？

3. **异步处理**  
   - 将长时间运行的 Agent 循环改为异步，避免阻塞。  
   - 实操：将 Agent 的核心处理逻辑改为 `async` 函数，使用 `asyncio.to_thread` 运行同步的工具函数，或直接使用异步数据库客户端。在 FastAPI 中使用 `async def` 端点。  
   ❓ **思考题**：如果某个工具是同步的且耗时较长（如文件读写），使用 `asyncio.to_thread` 会启动新线程，这对并发性能有何影响？是否应该考虑将工具也改为异步？

4. **可观测性**  
   - 学习记录日志、追踪（Tracing）和性能监控。  
   - 实操：集成 `loguru` 记录每次请求、工具调用、关键决策；使用 `opentelemetry` 或简单打点统计响应时间和 token 消耗。可选：将指标暴露给 Prometheus。  
   ❓ **思考题**：日志中可能包含用户隐私（如用户输入、记忆内容），你应该如何脱敏？对于敏感信息，是直接不记录还是加密存储？

5. **部署选项**  
   - 学习容器化：编写 Dockerfile 打包应用。  
   - 实操：创建 Dockerfile，将 FastAPI 应用和依赖打包，使用 `gunicorn + uvicorn` 运行。测试本地容器运行。了解云部署的基本步骤（如部署到 Render、AWS ECS 等）。  
   ❓ **思考题**：容器化时，向量数据库（如 Chroma）的数据文件应该放在哪里？如何持久化？如果使用多副本部署，如何保证记忆数据一致性？
